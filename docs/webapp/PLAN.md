# CorridorKey Web App — MVP Implementation Plan

## Context

**Problem:** CorridorKey is a powerful open-source neural green screen keyer, but it only runs via CLI/wizard. The goal is to build a private internal web tool so a single user (Mutlu) can sign in, upload green screen footage + alpha hints, dispatch GPU jobs, monitor progress, and download results — all from a browser.

**Approach:** Build the web shell, job model, storage flow, and worker wrapper as separate layers around the existing CorridorKey engine. The engine code is NOT modified. The worker imports `CorridorKeyModule` directly and calls `engine.process_frame()`.

---

## Architecture

```
┌──────────────┐       ┌──────────────────┐       ┌────────────────┐
│  Next.js App │──────▶│     Supabase     │◀──────│   GPU Worker   │
│  (Frontend   │       │  - Auth          │       │  (Python on    │
│   + API)     │       │  - Postgres DB   │       │   Runpod)      │
│              │       │  - Storage (S3)  │       │                │
└──────────────┘       └──────────────────┘       └────────────────┘

Browser (anon key only) ──▶ server routes ──▶ signed upload URLs ──▶ Storage
Server routes (service_role) ──▶ DB reads/writes
Worker (service_role) ──▶ polls jobs table, downloads/uploads Storage
```

### Security Model
- **Browser:** Only ever sees `NEXT_PUBLIC_SUPABASE_URL` and `NEXT_PUBLIC_SUPABASE_ANON_KEY`
- **Next.js server routes:** Use `SUPABASE_SERVICE_ROLE_KEY` for DB writes and generating signed Storage URLs
- **Worker:** Uses `SUPABASE_SERVICE_ROLE_KEY` for all DB/Storage operations
- **The service_role key NEVER appears in client bundles or `NEXT_PUBLIC_*` vars**

## Job Statuses

```
draft → queued → preparing → processing → uploading → completed
  │        │         │            │            │
  └────────┴─────────┴────────────┴────────────┴──────▶ failed
```

- **draft** — job row created, file uploads in progress
- **queued** — both files uploaded, waiting for worker
- **preparing** — worker claimed job, downloading files + extracting frames
- **processing** — inference running frame-by-frame
- **uploading** — inference done, uploading outputs to Storage
- **completed** — all outputs uploaded, ready for preview/download
- **failed** — error at any stage (error_message populated)

---

## Upload Flow (Browser → Server → Signed URL → Storage)

1. Client calls `POST /api/jobs` → server creates job row with `status='draft'`, returns `job_id`
2. Client calls `POST /api/upload` with `{job_id, file_type: 'input', file_name}` → server generates signed upload URL → returns URL
3. Client uploads input video directly to Storage using the signed URL
4. Client repeats for alpha file
5. Client calls `PATCH /api/jobs` with `{id, input_storage_path, alpha_storage_path}` → server flips to `queued`
6. Worker picks it up on next poll

## Worker — CorridorKey Integration

The worker calls the existing engine directly. No model code is rewritten.

**Critical files reused:**
- `CorridorKeyModule/backend.py:205` — `create_engine(backend, device, img_size)` factory
- `CorridorKeyModule/inference_engine.py:87` — `process_frame()` method signature
- `clip_manager.py:611-733` — Frame loop pattern to replicate

**Worker pipeline:**
1. Create temp dir → download assets → extract video frames
2. Run inference frame-by-frame (mirrors `clip_manager.py:611-733`)
3. Upload outputs to Supabase Storage
4. Insert `job_files` rows → mark completed

**Atomic claiming:** Worker uses `UPDATE ... WHERE status='queued'` guard + `worker_id` + `last_heartbeat`. Stale jobs (heartbeat > 5min) are recovered on startup.

## Design Decisions

| Decision | Rationale |
|----------|-----------|
| Signed upload URLs (not direct browser Storage) | Service_role key never touches the browser |
| `job_files` as single source of truth | No `output_paths` on jobs table = no drift |
| Atomic claim + heartbeat + stale recovery | Prevents stuck jobs after crashes |
| Draft → queued upload flow | DB row before uploads, status only queued after files confirmed |
| Polling, not websockets | Single user, v1 simplicity |
| Video-only upload (no image sequences) | One file upload instead of hundreds |
| Single engine instance in worker | Model load takes 10-20s + 22.7GB VRAM |
| Comp PNGs for preview, EXR for download | PNGs render in browser; EXR needs specialized software |

## Future: Preview Video

Post-v1: worker generates `preview.mp4` from Comp PNGs using ffmpeg. High-value for judging motion. `job_files` schema already supports `file_type = 'preview'`.
